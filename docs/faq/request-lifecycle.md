### 请求生命周期

Nest 应用程序处理请求并生成响应，称为 **请求生命周期**。使用中间件、管道、守卫和拦截器时，追踪特定代码执行点可能变得困难，尤其是在全局、控制器级别和路由级别组件中。总的来说，请求将通过中间件到守卫，然后到拦截器，最后回到拦截器（在响应生成时）。

#### 中间件

中间件执行顺序特定。首先，Nest 运行全局绑定的中间件（例如 __INLINE_CODE_2__），然后运行 __LINK_15__，这些中间件是根据路径确定的。中间件按顺序执行，类似于 Express 中间件的方式。在中间件跨模块绑定的情况下，根模块绑定的中间件将首先运行，然后是模块绑定的中间件，顺序与模块添加到 imports 数组的顺序相同。

#### 守卫

守卫的执行顺序为：全局守卫、控制器守卫、路由守卫。与中间件一样，守卫按绑定顺序执行。例如：

```bash
$ npm i -g @nestjs/cli
$ nest new project

```

__INLINE_CODE_3__ 将在 __INLINE_CODE_4__ 之前执行，所有三个都在 __INLINE_CODE_5__ 之前执行。

> info **提示**当讨论全局绑定的守卫 versus 控制器或本地绑定的守卫时，区别在于守卫的绑定位置。如果您使用 __INLINE_CODE_6__ 或在模块中提供组件，則是全局绑定。如果您在控制器类之前使用装饰器，则是控制器绑定；否则，如果您在路由声明之前使用装饰器，则是路由绑定。

#### 拦截器

拦截器基本上遵循守卫的模式，只有一点：拦截器返回 __LINK_16__，observable 将在先进后出顺序解决。因此，入站请求将通过标准的全局、控制器、路由级别解决，但响应侧的请求（即从控制器方法处理器返回后）将在路由、控制器、全局顺序解决。另外，任何由管道、控制器或服务抛出的错误都可以在拦截器的 __INLINE_CODE_7__ 操作符中读取。

#### 管道

管道遵循标准的全局到控制器到路由绑定顺序，同样在 __INLINE_CODE_8__ 参数方面为先进先出。但是在路由参数级别，如果您有多个管道运行，它们将在最后一个参数的管道到第一个参数的顺序运行。这也适用于路由级别和控制器级别的管道。例如，如果我们有以下控制器：

```bash
$ npm install --save hbs

```

那么 __INLINE_CODE_9__ 将在 `hbs`、`public`、`views` 之前运行，然后移动到 `hbs`，它遵循同样的顺序。如果有参数专属管道，它们将在控制器和路由级别管道运行后运行。

#### 过滤器

过滤器是唯一不解析全局的组件。相反，过滤器从最低级别开始执行，意味着执行从路由绑定的过滤器开始，然后是控制器级别、最后是全局过滤器。请注意，不能从过滤器到过滤器传递异常；如果路由级别过滤器捕获异常，控制器或全局过滤器不能捕获同一异常。要达到这种效果，可以使用过滤器继承。

> info **提示**过滤器只有在请求处理过程中遇到未捕获异常时才执行。如果捕获异常，例如使用 `views`，则不会触发 Exception Filters。一旦遇到未捕获异常，请求将跳过剩余的生命周期，直接跳到过滤器。

总的来说，请求生命周期如下：

1. 入站请求
2. 中间件

   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1. 全局守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器（pre-controller）
   - 4.1. 全局拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器
5. 管道
   - 5.1. 全局管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
   - 5.4. 路由参数管道
6. 控制器（方法处理器）
7. 服务（如果存在）
8. 拦截器（post-request）
   - 8.1. 路由拦截器
   - 8.2. 控制器拦截器
   - 8.3. 全局拦截器
