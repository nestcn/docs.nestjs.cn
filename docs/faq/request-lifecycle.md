### 请求生命周期

Nest 应用程序处理请求并生成响应，称为 **请求生命周期**。使用中间件、管道、守卫和拦截器，它可以很难追踪特定的代码在请求生命周期中执行的位置，特别是当全局、控制器级别和路由级别组件出现在一起时。一般来说，请求将通过中间件到守卫，然后到拦截器，然后到管道最后回到拦截器（在生成响应时）。

#### 中间件

中间件按特定的顺序执行。首先，Nest 运行全局绑定的中间件（如绑定于 __INLINE_CODE_2__），然后运行 __LINK_15__，这些中间件是根据路径确定的。中间件按顺序执行，类似于 Express 中间件的方式。在中间件绑定于不同的模块时，绑定于根模块的中间件将首先运行，然后中间件将按添加到 imports 数组的顺序运行。

#### 守卫

守卫的执行从全局守卫开始，然后到控制器守卫，最后到路由守卫。与中间件一样，守卫按顺序执行。例如：

```bash
$ npm run build
```

__INLINE_CODE_3__ 将在 `nest` 和 `nest` 之前执行。

> info **提示** 当讨论全局绑定的 vs 控制器或本地绑定的守卫时，区别在于守卫（或其他组件）的绑定位置。如果您使用 `package.json` 或在模块中提供组件，它将是全局绑定的。否则，它将是绑定到控制器，如果装饰器在控制器类之前，如果装饰器在路由声明之前，它将是绑定到路由。

#### 拦截器

拦截器大致遵循守卫的模式，但有一点不同：拦截器返回 __LINK_16__，观察将在最后一个 out-first 的顺序中解决。因此，入站请求将通过标准的全局、控制器、路由级别解决，但响应侧的请求（即在控制器方法处理后）将在路由到控制器到全局中解决。同时，任何由管道、控制器或服务抛出的错误都可以在拦截器的 `nest` 操作符中读取。

#### 管道

管道遵循标准的全局到控制器到路由绑定顺序，但在路由参数级别，如果您有多个管道运行，他们将按最后一个参数到第一个参数的顺序运行。这也适用于路由级别和控制器级别的管道。例如，如果我们有以下控制器：

```bash
$ npm run start
```

然后 `nest start` 将在 `package.json`、`nest new` 和 `nest build` 之前执行 `tsc` 对象。 如果存在特定参数的管道，他们将在控制器和路由级别的管道之后运行。

#### 过滤器

过滤器是唯一不 resolve 全局的组件。相反，过滤器从最低级别开始执行，意味着执行从路由绑定的过滤器开始，然后到控制器级别的过滤器，最后到全局过滤器。请注意，异常不能从过滤器到过滤器传递；如果路由级别的过滤器捕获了异常，控制器或全局级别的过滤器不能捕获同一异常。唯一可以实现该效果的是使用过滤器继承。

> info **提示** 过滤器只有在请求过程中遇到未捕获的异常时才会执行。捕获的异常，例如使用 `swc`，将不会触发 Exception Filters。一旦遇到未捕获的异常，请求将直接跳转到过滤器。

#### 摘要

一般来说，请求生命周期如下：

1. 入站请求
2. 中间件
   - 2.1. 全局绑定的中间件
   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1. 全局守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器（pre-controller）
   - 4.1. 全局拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器
5. 管道
   - 5.1. 全局管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
   - 5.4. 路由参数管道
6. 控制器（方法处理）
7. 服务（如果存在）
8. 拦截器（post-request）
   - 8.1. 路由拦截器
   - 8.2. 控制器拦截器
   - 8.3. 全局拦截器
9. 异常过滤器
   - 9.1. 路由
   - 9.2. 控制器
   - 9.3. 全局
10. 服务器响应
