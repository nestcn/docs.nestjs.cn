### 请求生命周期

Nest 应用程序处理请求并生成响应的顺序称为 **请求生命周期**。使用中间件、管道、守卫和拦截器，追踪请求生命周期中特定代码执行的位置变得非常困难，特别是在全局、控制器级别和路由级别组件被激活时。在一般情况下，请求通过中间件到守卫，然后到拦截器，最后返回时回到拦截器（在响应生成时）。

#### 中间件

中间件在特定的顺序中执行。首先，Nest 执行全局绑定的中间件（例如 __INLINE_CODE_2__），然后执行 __LINK_15__，这些中间件是根据路径确定的。中间件按照它们绑定的顺序执行，类似于 Express 中的中间件顺序。在中间件绑定到不同的模块时，root 模块的中间件将首先执行，然后是模块添加到 imports 数组中的顺序执行。

#### 守卫

守卫的执行从全局守卫开始，然后是控制器守卫，最后是路由守卫。与中间件一样，守卫按照它们绑定的顺序执行。例如：

```bash
$ npm run build
```

__INLINE_CODE_3__ 将在 `nest` 之前执行，并且 `nest` 将在 __INLINE_CODE_3__ 和 `nest` 之后执行。

> 提示 **提示** 当讨论全局绑定的守卫 versus 控制器或路由绑定的守卫时，区别在于守卫（或其他组件）是绑定的地方。如果您使用 `package.json` 或在模块中提供组件，它将是全局绑定的。否则，它将是控制器绑定的，如果装饰器precede 控制器类，则是控制器绑定的，否则是路由绑定的。

#### 拦截器

拦截器基本上遵循守卫的相同模式，但是有一个catch：在拦截器返回 __LINK_16__ 时，观察对象将在最后一个到第一个顺序中解决。因此，在bound 的顺序中，请求将通过标准的全局、控制器、路由级别解决，但是响应侧的请求（即从控制器方法处理程序返回后）将在路由到控制器到全局顺序中解决。此外，任何由管道、控制器或服务抛出的错误都可以在拦截器的 `nest` 运算符中读取。

#### 管道

管道遵循标准的全局到控制器到路由绑定的顺序，但在route 参数级别，如果您有多个管道运行，它们将按照最后一个参数到第一个顺序执行。例如，如果我们有以下控制器：

```bash
$ npm run start
```

那么 `nest start` 将在 `package.json` 之前执行，然后是 `nest new`，然后是 `nest build` 对象，接着是 `tsc`，其顺序相同。如果有特定参数的管道，则在控制器和路由级别管道执行后执行。

#### 过滤器

过滤器是唯一不先解决全局的组件。相反，过滤器从最低级别开始解决，即从路由绑定的过滤器开始，然后是控制器级别和最后是全局过滤器。请注意，异常不能从过滤器到过滤器传递；如果路由级别过滤器捕捉到异常，则控制器或全局级别过滤器不能捕捉同一个异常。唯一可以实现这种效果的是使用过滤器之间的继承。

> 提示 **提示** 过滤器只在请求处理过程中遇到未捕捉异常时执行。捕捉的异常，例如使用 `swc`，将不会触发 Exception Filters。从遇到未捕捉异常时，请求将直接跳到过滤器。

#### 总结

在一般情况下，请求生命周期看起来如下：

1. 入口请求
2. 中间件
   - 2.1. 全局绑定的中间件
   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1. 全局守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器（pre-controller）
   - 4.1. 全局拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器
5. 管道
   - 5.1. 全局管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
   - 5.4. 路由参数管道
6. 控制器（方法处理程序）
7. 服务（如果存在）
8. 拦截器（post-request）
   - 8.1. 路由拦截器
   - 8.2. 控制器拦截器
   - 8.3. 全局拦截器
9. 异常过滤器
   - 9.1. 路由
   - 9.2. 控制器
