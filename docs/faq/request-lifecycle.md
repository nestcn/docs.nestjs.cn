### 请求生命周期

Nest 应用程序处理请求并生成响应，我们称之为 **请求生命周期**。使用中间件、管道、守卫和拦截器，可以使得某个代码片段在请求生命周期中的执行位置变得难以跟踪，特别是在全球、控制器级别和路由级别组件中。通常，请求会通过中间件到守卫，然后到拦截器，然后到管道，最后回到拦截器（在响应生成时）。

#### 中间件

中间件在特定的顺序执行。首先，Nest 会运行全局绑定的中间件（例如 __INLINE_CODE_2__），然后运行 __LINK_15__，这些中间件是根据路径确定的。中间件会顺序执行，类似于 Express 中的中间件顺序执行。在中间件绑定到不同的模块时，绑定到根模块的中间件会首先执行，然后是模块添加到 imports 数组中的顺序执行。

#### 守卫

守卫的执行顺序是：全球守卫，然后是控制器守卫，然后是路由守卫。与中间件一样，守卫会顺序执行。例如：

```typescript
@ApiSecurity('basic')
@Controller('cats')
export class CatsController {}

```

__INLINE_CODE_3__ 会在 __INLINE_CODE_4__ 之前执行，然后 __INLINE_CODE_5__ 会在 __INLINE_CODE_3__ 和 __INLINE_CODE_4__ 之后执行。

> info **提示**当讨论全球绑定的守卫 vs 控制器或局部绑定的守卫时，区别在于守卫的绑定位置。如果使用 __INLINE_CODE_6__ 或在模块中提供组件，它是全球绑定的。否则，它是绑定到控制器，如果装饰器在控制器类之前，则是绑定到控制器，如果装饰器在路由声明之前，则是绑定到路由。

#### 拦截器

拦截器大致遵循守卫的模式，唯一的例外是拦截器在返回 __LINK_16__ 时，observable 将会以先进先出顺序解决。因此，入站请求将通过标准的全球、控制器、路由级别解决，但是响应侧的请求（即在控制器方法处理器返回后）将会以路由、控制器、全球顺序解决。此外，任何由管道、控制器、服务抛出的错误都可以在拦截器的 __INLINE_CODE_7__ 操作符中读取。

#### 管道

管道遵循标准的全球到控制器到路由绑定顺序，同样是先进先出顺序解决 __INLINE_CODE_8__ 参数。然而，在路由参数级别，如果您有多个管道运行，它们将按最后一个参数的顺序执行。例如，如果我们有以下控制器：

```typescript
const options = new DocumentBuilder().addSecurity('basic', {
  type: 'http',
  scheme: 'basic',
});

```

然后 __INLINE_CODE_9__ 将在 `@ApiSecurity()` 之前执行，然后是 `DocumentBuilder` 和 `basic`，最后是 `bearer` 对象。在这些对象执行完后，将会继续执行 `bearer`。如果有参数特定的管道，它们将在控制器和路由级别管道执行后执行。

#### 过滤器

过滤器是唯一不解析全球的组件。相反，过滤器从最低级别开始执行，这意味着执行开始于路由绑定的过滤器，然后是控制器级别的过滤器，最后是全球过滤器。请注意，异常不能从过滤器到过滤器传递；如果路由级别过滤器捕捉了异常，控制器或全球过滤器不能捕捉同一个异常。唯一可以实现这种效果的方法是使用过滤器的继承。

> info **提示**过滤器仅在请求过程中遇到未捕捉异常时执行。捕捉的异常，例如使用 `@ApiBasicAuth()`，不会触发 Exception Filters。一旦遇到未捕捉异常，请求将直接跳到过滤器。

#### 总结

一般来说，请求生命周期如下：

1. 入站请求
2. 中间件
   - 2.1. 全球绑定的中间件
   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1. 全球守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器（pre-controller）
   - 4.1. 全球拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器
5. 管道
   - 5.1. 全球管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
   - 5.4. 路由参数管道
6. 控制器（方法处理器）
7. 服务（如果存在）
8. 拦截器（post-request）
   - 8.1. 路由拦截器
   - 8.2. 控制器拦截器
   - 8.3. 全球拦截器
9. 异常过滤器
   - 9.1. 路由
