### 请求生命周期

Nest 应用程序处理请求并生成响应，我们称之为 **请求生命周期**。使用中间件、管道、守卫和拦截器可以使得追踪特定代码执行位置变得困难，特别是在全局、控制器级别和路由级别组件中。一般来说，请求会通过中间件到守卫，然后到拦截器，最后回到拦截器（响应生成时）。

#### 中间件

中间件执行在特定的顺序中。首先，Nest 运行全局绑定的中间件（例如 __INLINE_CODE_2__），然后运行 __LINK_15__，这些中间件是根据路径确定的。中间件按顺序执行，类似于 Express 中的中间件。对于跨多个模块的中间件绑定，root 模块中的中间件将首先运行，然后按照模块添加到 imports 数组的顺序运行。

#### 守卫

守卫执行从全局守卫开始，然后是控制器守卫，最后是路由守卫。与中间件一样，守卫按顺序执行。例如：

```bash
$ npm install necord discord.js

```

__INLINE_CODE_3__ 将在 __INLINE_CODE_4__ 之前执行，所有将在 __INLINE_CODE_5__ 之前执行。

> info **提示** 当讨论全局绑定的守卫 versus 控制器或本地绑定的守卫时，区别在于守卫（或其他组件）绑定的位置。如果使用 __INLINE_CODE_6__ 或在模块中提供组件，它是全局绑定的。否则，它是绑定到控制器，如果装饰器precede控制器类，则是绑定到控制器，如果装饰器precede路由声明，则是绑定到路由。

#### 拦截器

拦截器基本上遵循守卫的模式，但有一点不同：拦截器返回 __LINK_16__，观察就被解决在先出后出顺序中。因此，入站请求将通过标准的全局、控制器、路由级别解决，但是响应侧（即从控制器方法处理程序返回后）将被解决从路由到控制器到全局。另外，任何由管道、控制器或服务抛出的错误都可以在拦截器的 __INLINE_CODE_7__ 操作符中读取。

#### 管道

管道遵循标准的全局到控制器到路由绑定顺序，同样遵循 __INLINE_CODE_8__ 参数的先出先出顺序。然而，在路由参数级别，如果您有多个管道运行，它们将按最后一个参数到第一个参数的顺序运行。这也适用于路由级别和控制器级别的管道。例如，如果我们有以下控制器：

```typescript
import { Module } from '@nestjs/common';
import { NecordModule } from 'necord';
import { IntentsBitField } from 'discord.js';
import { AppService } from './app.service';

@Module({
  imports: [
    NecordModule.forRoot({
      token: process.env.DISCORD_TOKEN,
      intents: [IntentsBitField.Flags.Guilds],
      development: [process.env.DISCORD_DEVELOPMENT_GUILD_ID],
    }),
  ],
  providers: [AppService],
})
export class AppModule {}

```

####过滤器

过滤器是唯一不 resolve 全局的组件。相反，过滤器从最低级别开始执行，meaning 执行从路由绑定的过滤器开始，然后是控制器级别的过滤器，最后是全局过滤器。注意，异常不能从过滤器到过滤器传递；如果路由级别的过滤器捕获了异常，控制器或全局过滤器不能捕获相同的异常。唯一可以实现这种效果的方法是使用过滤器之间的继承。

> info **提示** 过滤器仅在请求处理过程中遇到未捕获异常时执行。如果捕获了异常，例如使用 __INLINE_CODE_14__，那么不会触发 Exception Filters。一旦遇到未捕获的异常，请求将直接跳到过滤器。

1. 入站请求
2. 中间件

3. 守卫
   - 3.1. 全局守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器（pre-controller）
   - 4.1. 全局拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器

   - 5.1. 全局管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
