### 请求生命周期

Nest 应用程序处理请求并生成响应，称为**请求生命周期**。通过中间件、管道、守卫和拦截器，可以追踪请求生命周期中特定代码执行的顺序，特别是在全局、控制器级别和路由级别组件中。一般来说，请求通过中间件到守卫，然后到拦截器，然后到管道最后返回到拦截器（在响应生成时）。

#### 中间件

中间件按特定的顺序执行。首先，Nest 运行全局绑定的中间件（如 __INLINE_CODE_2__），然后运行 __LINK_15__，这些中间件根据路径确定。中间件按绑定的顺序执行，类似于 Express 中的中间件。对于跨多个模块绑定的中间件，root 模块的中间件将先执行，然后是按 imports 数组添加的模块顺序执行。

#### 守卫

守卫执行始于全局守卫，然后继续到控制器守卫，最后到路由守卫。与中间件一样，守卫按绑定的顺序执行。例如：

```bash
$ npm install --save @nestjs/cqrs

```

__INLINE_CODE_3__ 将在 __INLINE_CODE_4__ 之前执行，并且都在 __INLINE_CODE_5__ 之前执行。

> 提示 **hint** 当 discussing 绑定的守卫vs控制器或局部绑定的守卫时，区别在于守卫（或其他组件）的绑定位置。如果您使用 __INLINE_CODE_6__ 或通过模块提供组件，它是全局绑定的。否则，它是绑定到控制器如果装饰器precede一个控制器类，或者绑定到路由如果装饰器precede一个路由声明。

#### 拦截器

拦截器遵循守卫的同一模式，只是有一点不同：拦截器返回 __LINK_16__，observable 将在 first-in-last-out 顺序中解决。因此，在bound 的顺序中，请求将通过标准的全局、控制器、路由级别解决，但是响应侧（即在控制器方法处理程序返回后）将在路由到控制器到全局顺序中解决。此外，任何由管道、控制器或服务抛出的错误可以在拦截器的 __INLINE_CODE_7__ 运算符中读取。

#### 管道

管道遵循标准的全局到控制器到路由绑定顺序，在 __INLINE_CODE_8__ 参数方面同样是 first-in-first-out 顺序。然而，在路由参数级别，如果您有多个管道运行，它们将按最后一个参数到第一个参数的顺序执行。例如，如果我们有以下控制器：

```typescript
import { Module } from '@nestjs/common';
import { CqrsModule } from '@nestjs/cqrs';

@Module({
  imports: [CqrsModule.forRoot()],
})
export class AppModule {}

```

#### 过滤器

过滤器是唯一不 resolve 全局的组件。相反，过滤器从最低级别开始执行，即从路由绑定的过滤器开始，然后是控制器级别，然后是全局过滤器。请注意，异常不能从过滤器到过滤器传递；如果路由级别过滤器捕捉异常，控制器或全局级别过滤器不能捕捉同一个异常。唯一可以实现这种效果的是使用过滤器之间的继承。

> 提示 **hint** 过滤器只有在请求过程中遇到未捕捉异常时才执行。如果捕捉的异常，例如使用 __INLINE_CODE_14__，不会触发 Exception Filters。一旦遇到未捕捉异常，请求将跳过剩余的生命周期并直接到过滤器。

#### 概要

在一般情况下，请求生命周期如下：

1. incoming 请求
2. 中间件
   - 2.1. 全局绑定的中间件
   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1. 全局守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器（pre-controller）
   - 4.1. 全局拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器
5. 管道
   - 5.1. 全局管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
   - 5.4. 路由参数管道
6. 控制器（method handler）
7. 服务（如果存在）
8. 拦截器（post-request）
   - 8.1. 路由拦截器
   - 8.2. 控制器拦截器
   - 8.3. 全局拦截器
9. 异常过滤器
   - 9.1. 路由
   - 9.2. 控制器
   - 9.3.
