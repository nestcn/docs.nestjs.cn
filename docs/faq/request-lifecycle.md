### 请求生命周期

Nest 应用程序处理请求并生成响应，这是一个称为 **请求生命周期** 的顺序。使用中间件、管道、守卫和拦截器时，追踪特定代码在请求生命周期中执行的位置可能会变得很困难，特别是在全球、控制器级别和路由级别组件出现时。在一般情况下，请求将通过中间件流向守卫，然后流向拦截器，然后流向管道，并最终在返回路径中流向拦截器（在生成响应时）。

#### 中间件

中间件在特定顺序执行。首先，Nest 执行全球绑定的中间件（例如绑定到 __INLINE_CODE_2__ 等），然后执行 __LINK_15__，这些中间件是根据路径确定的。中间件按顺序执行，类似于 Express 中间件的方式。在中间件绑定到不同模块时，绑定到根模块的中间件将首先执行，然后是模块绑定的中间件顺序。

#### 守卫

守卫的执行从全球守卫开始，然后是控制器守卫，最后是路由守卫。与中间件类似，守卫按顺序执行。例如：

```bash
$ npm run build
```

__INLINE_CODE_3__ 将在 `nest` 和 `nest` 之前执行。

> info **提示** 在 discuss 关于全球绑定的守卫 vs 控制器或局部绑定的守卫时，区别在于守卫（或其他组件）被绑定的位置。如果您使用 `package.json` 或在模块中提供组件，它将是全球绑定的。否则，它将是绑定到控制器，如果装饰器在控制器类之前出现，或者绑定到路由，如果装饰器在路由声明之前出现。

#### 拦截器

拦截器大致遵循守卫的模式，但有一点不同：拦截器返回 __LINK_16__ 时，observable 将在最后一个在第一出执行。因此，入站请求将通过标准的全球、控制器、路由级别解决方案，但是响应侧(request) 将在路由到控制器到全球解决方案中。另外，任何由管道、控制器或服务抛出的错误都可以在拦截器的 `nest` 操作符中读取。

#### 管道

管道遵循标准的全球到控制器到路由绑定的顺序，同样，管道的参数将在最后一个在第一出执行。然而，在路由参数级别，如果您有多个管道运行，它们将在最后一个参数的 pipe 到第一个参数的顺序中执行。这同样适用于路由级别和控制器级别的管道。例如，如果我们有以下控制器：

```bash
$ npm run start
```

那么 `nest start` 将在 `package.json`、`nest new` 和 `nest build` 对象之前执行，然后是 `tsc` 对象。如果存在特定参数的管道，它们将在控制器和路由级别的管道执行后执行。

#### 过滤器

过滤器是唯一不 resolve 全球的组件。相反，过滤器从最低级别开始执行，意味着执行从路由绑定的过滤器开始，然后是控制器级别的过滤器，最后是全球过滤器。请注意，不能将异常从过滤器传递到过滤器；如果路由级别的过滤器捕捉了异常，控制器或全球级别的过滤器不能捕捉同一个异常。要实现这种效果，您需要使用过滤器之间的继承。

> info **提示** 过滤器只有在请求过程中遇到未捕捉的异常时才执行。如果捕捉了异常，例如使用 `swc`，那么 Exception Filters 将不会触发。一旦遇到未捕捉的异常，request 将跳过剩余的生命周期并直接跳到过滤器。

#### 综述

总的来说，请求生命周期如下：

1. 入站请求
2. 中间件
   - 2.1. 全球绑定的中间件
   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1. 全球守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器 (pre-controller)
   - 4.1. 全球拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器
5. 管道
   - 5.1. 全球管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
   - 5.4. 路由参数管道
6. 控制器 (方法处理程序)
7. 服务 (如果存在)
8. 拦截器 (post-request)
   - 8.1. 路由拦截器
   - 8.2. 控制器拦截器
   - 8.3. 全球拦截器
9. 异常过滤器
