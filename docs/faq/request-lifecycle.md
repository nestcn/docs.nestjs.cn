### 请求生命周期

Nest 应用程序处理请求并生成响应，我们称之为 **请求生命周期**。使用中间件、管道、守卫和拦截器，可以追踪某个代码块在请求生命周期中的执行位置，特别是全局、控制器级别和路由级别组件的组合。在一般情况下，请求会通过中间件到守卫，然后到拦截器，然后到管道，并最终返回到拦截器（在响应生成时）。

#### 中间件

中间件将在特定的顺序中执行。首先，Nest 将运行全局绑定的中间件（例如 __INLINE_CODE_2__），然后运行 __LINK_15__，这些中间件在路径上被确定。中间件将顺序地运行，如 Express 中间件一样。在中间件绑定到不同模块时，绑定到根模块的中间件将首先运行，然后绑定到模块中的中间件将按照模块添加到 imports 数组的顺序运行。

#### 守卫

守卫的执行顺序是：首先是全局守卫，然后是控制器守卫，最后是路由守卫。与中间件一样，守卫将按照绑定的顺序运行。例如：

```typescript
export enum Role {
  User = 'user',
  Admin = 'admin',
}

```

#### 拦截器

拦截器的大致模式与守卫相似，但有一点不同：拦截器返回 __LINK_16__，observable 将在最后一个出现在前一个顺序中解决。因此，入站请求将通过标准的全局、控制器、路由级别解决，但响应侧（即在控制器方法处理程序返回后）将在路由到控制器到全局顺序中解决。另外，任何由管道、控制器或服务抛出的错误都可以在拦截器的 __INLINE_CODE_7__ 运算符中读取。

#### 管道

管道将按照标准的全局到控制器到路由绑定顺序执行，但在路由参数级别，如果您有多个管道运行，它们将按照最后一个参数到第一个参数的顺序运行。例如，如果我们有以下控制器：

```typescript
import { SetMetadata } from '@nestjs/common';
import { Role } from '../enums/role.enum';

export const ROLES_KEY = 'roles';
export const Roles = (...roles: Role[]) => SetMetadata(ROLES_KEY, roles);

export const ROLES_KEY = 'roles';
export const Roles = (...roles) => SetMetadata(ROLES_KEY, roles);
```

#### 过滤器

过滤器是唯一不解析全局的组件。相反，过滤器从最低级别开始解析，这意味着执行从路由绑定的过滤器开始，然后是控制器级别的过滤器，最后是全局过滤器。请注意，不能从过滤器到过滤器传递异常。如果路由级别的过滤器捕捉了异常，控制器或全局级别的过滤器不能捕捉相同的异常。唯一可以实现这种效果的是使用过滤器继承。

> info **提示** 过滤器只有在请求过程中遇到未捕捉异常时才会执行。如果捕捉了异常，例如使用 __INLINE_CODE_14__，则不会触发异常过滤器。一旦遇到未捕捉的异常，请求将直接跳到过滤器。

在一般情况下，请求生命周期如下：

1. 入站请求
2. 中间件

   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1. 全局守卫
   - 3.2. 控制器守卫
   - 3.3. 路由守卫
4. 拦截器（pre-controller）
   - 4.1. 全局拦截器
   - 4.2. 控制器拦截器
   - 4.3. 路由拦截器

   - 5.1. 全局管道
   - 5.2. 控制器管道
   - 5.3. 路由管道
   - 5.4. 路由参数管道
6. 控制器（方法处理程序）
7. 服务（如果存在）

   - 8.2. 控制器拦截器
