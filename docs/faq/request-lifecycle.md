### 请求生命周期

Nest 应用程序处理请求并生成响应，称为 **请求生命周期**。使用中间件、管道、守卫和拦截器，可以追踪特定代码在请求生命周期中的执行位置，特别是全局、控制器级别和路由级别组件的参与。在一般情况下，请求将通过中间件到守卫，然后到拦截器，最后回到拦截器（响应生成时）。

#### 中间件

中间件在特定顺序执行。首先，Nest 执行全局绑定的中间件（例如 `npx`），然后执行 __LINK_15__，这些中间件根据路径确定。中间件顺序执行，类似于 Express 中的中间件顺序。对于跨多个模块绑定的中间件，root 模块中的中间件将先执行，然后按照模块添加到 imports 数组的顺序执行。

#### 守卫

守卫执行从全局守卫开始，然后是控制器守卫，最后是路由守卫。与中间件一样，守卫按绑定顺序执行。例如：

```bash
$ npm i -D @compodoc/compodoc

```

__INLINE_CODE_3__ 将在 __INLINE_CODE_4__ 之前执行，并且都在 __INLINE_CODE_5__ 之前执行。

> 信息 **提示** 当讨论全球绑定的守卫 vs 控制器或本地绑定的守卫时，区别在于守卫（或其他组件）绑定的位置。如果您使用 __INLINE_CODE_6__ 或在模块中提供组件，它将是全球绑定的。否则，它将绑定到控制器，如果装饰器前置控制器类，则绑定到路由。

#### 拦截器

拦截器大致遵循守卫的模式，唯一的区别是拦截器返回 __LINK_16__，观察项将在最后一个出现在最前面。因此， incoming 请求将通过标准的全球、控制器、路由级别解决方案，但是响应侧（即返回控制器方法处理器后）将在路由、控制器和全球级别中解决。另外，任何由管道、控制器或服务抛出的错误都可以在拦截器的 __INLINE_CODE_7__ 操作符中读取。

#### 管道

管道遵循标准的全球到控制器到路由绑定的顺序，同样地，对于 __INLINE_CODE_8__ 参数的顺序是“先出先入”。然而，在路由参数级别，如果您有多个管道运行，它们将按最后一个参数到第一个参数的顺序执行。例如，如果我们有以下控制器：

```bash
$ npx @compodoc/compodoc -p tsconfig.json -s

```

那么 __INLINE_CODE_9__ 将在 __INLINE_CODE_10__ 之前执行，然后在 __INLINE_CODE_11__ 之前执行，最后在 __INLINE_CODE_12__ 之前执行。如果存在特定参数的管道，它们将在控制器和路由级别管道执行后执行。

#### 过滤器

过滤器是唯一不 resolve 全球的组件。相反，过滤器从最低级别开始执行，这意味着执行从路由绑定的过滤器开始，然后是控制器级别，最后是全球级别。请注意，不能从过滤器到过滤器传递异常；如果路由级别过滤器捕获异常，则控制器或全球级别过滤器不能捕获同一个异常。唯一可以实现这个效果的是使用过滤器之间的继承关系。

> 信息 **提示** 过滤器仅在请求处理过程中遇到未捕获的异常时执行。如果捕获的异常（例如使用 __INLINE_CODE_14__），则不会触发 Exception Filters。一旦遇到未捕获的异常，请求将跳过剩余的生命周期，直接跳转到过滤器。

#### 总结

一般来说，请求生命周期如下：

1. Incoming 请求
2. 中间件
   - 2.1. 全局绑定的中间件
   - 2.2. 模块绑定的中间件
3. 守卫
   - 3.1 全局守卫
   - 3.2 控制器守卫
   - 3.3 路由守卫
4. 拦截器（pre-controller）
   - 4.1 全局拦截器
   - 4.2 控制器拦截器
   - 4.3 路由拦截器
5. 管道
   - 5.1 全局管道
   - 5.2 控制器管道
   - 5.3 路由管道
   - 5.4 路由参数管道
6. 控制器（方法处理器）
7. 服务（如果存在）
8. 拦截器（post-request）
   - 8.1 路由拦截器
   - 8.2 控制器拦截器
   - 8.3 全局拦截器
9. 异常过滤器
   - 9.1 路由
   - 9.2 控制器
   - 9.3 全球
10. 服务器响应
